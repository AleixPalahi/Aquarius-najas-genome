Gene body coverage

Checking gene body coverage in order to test for 3' bias. If there are degraded transcripts you miss the 5' end, more frequent reads towards the 3' end (if the sequencing protocol includes polyA capture).

Had to generate a bed file using AGAT v1.3.2 agat_convert_sp_gff2bed.pl -gff braker.gff -o braker.bed (gff done for annotation QC)

To get the report I used RSeQC's geneBody_coverage.py (rseqc v 2.6.4)

Functional annotation

I will try I) InterProScan and II) BLAST+ as presented in this tutorial by NBIS. On top of that, I will also run III) eggnog-mapper

I had to remove stop codons from the braker output as some tools (InterProScan) have an issue with the asterisk. sed -e 's/*//g' braker.aa > braker_nostop.faa

I) InterProScan v5.62-94.0

I ran this with similar settings as shown in the NBIS tutorial, but I skipped the -appl flag to expand the search to all supported databases (CDD,Coils,FunFam,Gene3D,Hamap,MobiDBLite,NCBIfam,PANTHER,Pfam,Phobius,PIRSF,PRINTS,ProSitePatterns,ProSiteProfiles,SFLD,SignalP_EUK,SignalP_GRAM_NEGATIVE,SignalP_GRAM_POSITIVE,SMART,SUPERFAMILY,TMHMM).

It ran relatively fast for ~15h resulting in 353 923 hits.

II) BLAST+ v2.16.0+

I ran blastp twice, once against Arthropods (taxid: 6656) and a second time limiting the search to Hemiptera (taxid: 7524). I use the uniprot database, limit the output to max 10 target sequences per query (limiting to one hit may not necessarily give only the best hit), I chose a comparably "relaxed" e-value of 1e-3, I want to be a bit more broad with these settings as blast is already more precise than other tools and I can always filter hits afterwards if needed.

BLAST is of course much slower, even when running on 16 threads.

Had to build my own database, Uppmax doesn't store the fasta file for uniprot

III) Eggnog-mapper 2.1.9 (eggNOG DB v 5.0.2)

Next, I also ran eggnog-mapper with three different settings, a) using mmseqs2 v12-113e3 with default sensitivity settings, b) mmseqs2 with high sensitivity settings and c) using Diamond v2.0.11

Opinions on which tools is better are torn so I wanted to try both. mmseqs2 is recommended over diamond when dealing with full genomes/scaffolds which isn't an issue for my input. Each creator argues - as usual - that their tool outperforms the other.

For the first mmseqs2 run, I use default settings, 3 iterations, starting at sensitivity s=3 and ending with s=7 (2nd iteration at s=5), I took the maximum value from their most recent paper which claims to be more accurate than Diamond. I limit the search to only Arthropods (taxid: 6656).

The second mmseqs2 run was more sensitive, again going through 3 iterations, but starting at s=3.5 and ending at s=8.5 (2nd iteration at s=6) and again limiting the search to Arthropods.

For Diamond, I use multiple iterations with increasing sensitivity in order to kepp it consistent with mmseqs2's approach. I chose the highest setting for final sensitivity (ultra-sensitive) as the authors claim that it then becomes more accurate than mmseqs2 at high sensitivity (s=7.5). I am using BLOSUM62 as substitution matrix as this is also mmseqs2's default. I also limited the search to Arthropods here.

Eggnog runs much faster than the other two tools, on default settings it took mm2seqs2 ~43 mins, on high-sensitivity ~52 mins and using Diamond ~53 mins.

The results summarised:

Approach N_hits N_annotations
mmseqs2 def 18 621 15 936
mmseqs2 HS 18 640 15 942

Diamond picked up slightly more unique features and genes, but some were also unique to mmseqs2 (see eval_annot_methods.html for comparisons). I decided to merge both diamond and mmseqs2 (high sensitivity) outputs.

Filtering:

Some publications that I screened used minimum ORF size, premature stop codons, incomplete start/stop codons and gene length.

1) Minimum ORF size agat_sp_filter_by_ORF_size

I'll try the default of > 100 aa as well as > 50 aa first. Afterwards, I also added > 30 aa.

50 aa cutoff removes 627 transcripts (3.011% of all transcripts), 65 genes and 68 genes have at least one transcript removed (0.445% and 0.466% of all genes respectively). 100 aa cutoff removes 1614 transcripts (7.75%), 915 (6.265%) genes and 1010 (6.915%) genes with at least one removed transcript. I also tried 30 aa which removed 605 transcripts (2.905%), 46 genes (0.315%) and 46 genes that had at least one transcript removed. Among the publications I came across, people used 30 - 50 aa as cutoff, so I would stick with the 50 aa threshold. Even at 50 aa, all excluded genes are unnamed (identified mostly by GeneMark, some by gmst)

2) Distance between loci agat_sp_filter_by_locus_distance

I'm not sure how important this filter, as I understand this is more important for training specific tools that require "clean" intergenic regions. The main issue are just overlapping genes. No default given, this tutorial by the AGAT creator uses 500 bp. It's also possible to run this command without filtering and just adding a low_dist flag including the actual distance. First I will just see how many features I lose.

Using 500 bp, the filter throws out 2623 loci (17.972%) this seems quite harsh and since I saw that no genes are overlapping, I would skip this filter.

3) Gene length agat_sp_filter_gene_by_length

I've only come across people using a minimum filter, not maximum.

I'll filter by maximum size using the 95% (< 34140) and 90% (<= 22029) quantile. I obtained the quantiles through R quantiles using a type 1 distribution (discontinuous, inverse of empirical distribution function) as this seems to overlap with AGAT's calculation. Lengths for the actual filtering are determined by concatenating all exons of a gene together, so the quantiles themselves are not fully accurate. Wouldn't this also then only make sense after removing isoforms? I'll also try to filter by minimum length using the same setting as minimum ORF size to see how much these filters overlap (using 150 [50*3] and 300 [100*3 bp]).

The minimum filter worked fine and removed 62 (0.425%) or 866 (5.934%) of genes when using 150 bp or 300 bp as cutoffs, respectively. I also tried >= 50 bp which I came across in two references, this removes just 23 genes (0.157%). No genes are exclusive to this filter compared to the ORF filter, just using the ORF is fine then.

4) Incomplete cds (start and/or stop codon missing) agat_sp_filter_incomplete_gene_coding_models

I have some start and stop codons that are shorter than 3 bp (0.389% and 0.101% respectively), I'm not sure if this filter can remove these. This one can also be run as a flag rather than a filter.

Two genes (rest-a [transcriptional repressor of neural genes in non-neural cells] and Grik2 [synaptic plasticity, learning, and memory]) were missing a start codon.

5) Merge overlapping genes into a single one agat_sp_fix_overlaping_genes

Turns overlapping genes into isoforms.

No overlapping genes were detected.

6) Add "pseudo" to genes with premature stop codons agat_sp_flag_premature_stop_codons

This just adds a flag and shouldn't remove anything, still good to know how many features would get flagged.

No feature had premature stop codons.

In the end, for final filtering, I only implemented a filter for minimum ORF size (>= 50 aa) and throwing out incomplete CDS
